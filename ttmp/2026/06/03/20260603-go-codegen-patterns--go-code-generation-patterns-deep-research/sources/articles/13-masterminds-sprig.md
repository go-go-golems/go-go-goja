Sprig is a library that provides more than 100 commonly used template functions.</p> <p dir="auto">It is inspired by the template functions found in <a href="http://twig.sensiolabs.org/documentation">Twig</a> and in various JavaScript libraries, such as <a href="http://underscorejs.org/">underscore.js</a>.</p> <H2 dir="auto">IMPORTANT NOTES</H2> <p dir="auto">Sprig leverages <a href="https://github.com/imdario/mergo">mergo</a> to handle merges. In its v0.3.9 release, there was a behavior change that impacts merging template functions in sprig. It is currently recommended to use v0.3.10 or later of that package. Using v0.3.9 will cause sprig tests to fail.</p> <H2 dir="auto">Package Versions</H2> <p dir="auto">There are two active major versions of the <code>sprig</code> package.</p> <ul dir="auto"> <li>v3 is currently stable release series on the <code>master</code> branch. The Go API should remain compatible with v2, the current stable version. Behavior change behind some functions is the reason for the new major version.</li> <li>v2 is the previous stable release series. It has been more than three years since the initial release of v2. You can read the documentation and see the code on the <a href="https://github.com/Masterminds/sprig/tree/release-2">release-2</a> branch. Bug fixes to this major version will continue for some time.</li> </ul> <H2 dir="auto">Usage</H2> <p dir="auto"><strong>Template developers</strong>: Please use Sprig's <a href="http://masterminds.github.io/sprig/">function documentation</a> for detailed instructions and code snippets for the &gt;100 template functions available.</p> <p dir="auto"><strong>Go developers</strong>: If you'd like to include Sprig as a library in your program, our API documentation is available <a href="http://godoc.org/github.com/Masterminds/sprig">at GoDoc.org</a>.</p> <p dir="auto">For standard usage, read on.</p> <H3 dir="auto">Load the Sprig library</H3> <p dir="auto">To load the Sprig <code>FuncMap</code>:</p> <pre><code>import (
  "github.com/Masterminds/sprig/v3"
  "html/template"
)

// This example illustrates that the FuncMap *must* be set before the
// templates themselves are loaded.
tpl := template.Must(
  template.New("base").Funcs(sprig.FuncMap()).ParseGlob("*.html")
)</code></pre> <H3 dir="auto">Calling the functions inside of templates</H3> <p dir="auto">By convention, all functions are lowercase. This seems to follow the Go idiom for template functions (as opposed to template methods, which are TitleCase). For example, this:</p> <div><pre><code>{{ "hello!" | upper | repeat 5 }}</code></pre></div> <p dir="auto">produces this:</p> <div><pre><code>HELLO!HELLO!HELLO!HELLO!HELLO!</code></pre></div> <H2 dir="auto">Principles Driving Our Function Selection</H2> <p dir="auto">We followed these principles to decide which functions to add and how to implement them:</p> <ul dir="auto"> <li>Use template functions to build layout. The following types of operations are within the domain of template functions: <ul dir="auto"> <li>Formatting</li> <li>Layout</li> <li>Simple type conversions</li> <li>Utilities that assist in handling common formatting and layout needs (e.g. arithmetic)</li> </ul> </li> <li>Template functions should not return errors unless there is no way to print a sensible value. For example, converting a string to an integer should not produce an error if conversion fails. Instead, it should display a default value.</li> <li>Simple math is necessary for grid layouts, pagers, and so on. Complex math (anything other than arithmetic) should be done outside of templates.</li> <li>Template functions only deal with the data passed into them. They never retrieve data from a source.</li> <li>Finally, do not override core Go template functions.</li> </ul> </article>

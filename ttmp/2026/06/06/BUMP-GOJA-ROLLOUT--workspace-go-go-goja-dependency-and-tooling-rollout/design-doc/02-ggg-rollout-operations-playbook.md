# GGG Rollout Operations Playbook

## Executive Summary

Use the installed `ggg` CLI as the operator control plane for large go-go-golems rollouts. The intended workflow is: inventory repositories, capture a rollout YAML config, validate every target with `GOWORK=off`, push branches and open PRs, let the PR-open automation start Codex, watch PR readiness in batch, merge in dependency order with merge commits, verify post-merge GitHub Actions, and only then publish upstream tags or continue downstream.

This playbook turns the infra-tooling documentation into a copy/paste-ready procedure for the `BUMP-GOJA-ROLLOUT` style of work. It assumes code changes are made per repository and committed locally, and focuses on the PR/review/release-operations layer.

## Problem Statement

Manual multi-repository rollouts have three recurring failure modes:

1. **Workspace leakage:** local `go.work` makes downstream builds pass even when the published module graph is broken.
2. **Review/CI drift:** PRs are opened across many repos, but Codex status, failed checks, stale reviews, and partial readiness are tracked by hand.
3. **Dependency-order mistakes:** downstream PRs are merged or tagged before required upstream commits are merged and published.

`ggg` exists to make those operations explicit, structured, and repeatable.

## Non-Negotiable Rules

1. **Use `GOWORK=off` for downstream validation.** Do not trust workspace-linked green builds.
2. **Open PRs; do not push source changes directly to `main`.** Tags may be pushed by release tooling after the commit is on `main`.
3. **Merge with real merge commits.** Use `gh pr merge --merge --delete-branch`; do not squash release-train PRs.
4. **Merge/publish in dependency order.** Early downstream PRs are allowed for feedback, but do not merge them until upstream tags are available and downstream `GOWORK=off` validation passes.
5. **Treat Codex feedback as a blocker.** Green Actions are not enough if `ggg pr ready` reports current-head Codex feedback.
6. **Ignore only known-noisy CI workflows intentionally.** Example: `--ignore-workflow "Secret Scanning"` when it is known to be non-blocking.

## Rollout Workspace Layout

Recommended ticket-local files:

```text
ttmp/.../BUMP-GOJA-ROLLOUT--.../
  scripts/
    rollout.yaml          # ggg rollout config
    prs.yaml              # generated by ggg rollout push-prs
    actions.yaml          # optional post-merge CI manifest
  reference/
    01-diary.md           # chronological work diary
    02-ggg-rollout-research-logbook.md
  design-doc/
    02-ggg-rollout-operations-playbook.md
```

Keep PR lists and generated reports in the ticket so another operator can resume without reconstructing state from shell history.

## Phase 1: Inventory the Workspace

From the rollout workspace root:

```bash
WORKSPACE=/home/manuel/workspaces/2026-06-06/bump-goja

# Find all cloned Go repos that require go-go-goja.
ggg rollout inventory \
  --root "$WORKSPACE" \
  --require-module github.com/go-go-golems/go-go-goja \
  --output table

# Save machine-readable evidence if useful for the ticket.
ggg rollout inventory \
  --root "$WORKSPACE" \
  --require-module github.com/go-go-golems/go-go-goja \
  --output json \
  --output-file ttmp/.../scripts/rollout-inventory.json
```

Use the inventory to decide dependency order. Do not rely on memory.

## Phase 2: Create the Rollout Config

Create a first config with included repos and validation commands:

```bash
ggg rollout init \
  --id BUMP-GOJA-ROLLOUT \
  --name "Bump go-go-goja to v0.8.3" \
  --workspace "$WORKSPACE" \
  --branch task/bump-goja \
  --include "plz-confirm,go-go-os-chat,go-go-app-inventory,smailnail" \
  --validation "GOWORK=off go build ./..." \
  --validation "GOWORK=off go test ./..." \
  --write-to ttmp/.../scripts/rollout.yaml
```

For repositories with special validation, either use a separate rollout config or edit `validation.commands` manually. Example for `smailnail`:

```yaml
validation:
  commands:
    - name: build
      run: GOWORK=off go build ./...
    - name: test-sqlite-fts5
      run: GOWORK=off go test -tags sqlite_fts5 ./...
```

Important fields in `rollout.yaml`:

```yaml
id: BUMP-GOJA-ROLLOUT
name: Bump go-go-goja to v0.8.3
workspace: /home/manuel/workspaces/2026-06-06/bump-goja
branch: task/bump-goja
base: origin/main
commit_message: Bump go-go-goja to v0.8.3
selection:
  require_go_mod_contains: []
  include:
    - plz-confirm
    - go-go-os-chat
    - go-go-app-inventory
  exclude: []
validation:
  commands:
    - name: build
      run: GOWORK=off go build ./...
    - name: test
      run: GOWORK=off go test ./...
  continue_on_error: true
  log_dir: .ggg-rollout-logs
pull_request:
  title: Bump go-go-goja to v0.8.3
  body_file: ttmp/.../scripts/pr-body.md
  output_prs: ttmp/.../scripts/prs.yaml
  no_verify_push: false
readiness:
  trigger_codex: false
  watch: false
```

## Phase 3: Local Validation

Dry-run first:

```bash
ggg rollout validate ttmp/.../scripts/rollout.yaml --dry-run --output table
```

Run validation:

```bash
ggg rollout validate ttmp/.../scripts/rollout.yaml --output table
```

If validation fails, inspect logs under the configured `validation.log_dir`. Fix and commit per repository before opening PRs.

For one-off manual validation, keep the command explicit:

```bash
cd "$WORKSPACE/go-go-app-inventory"
GOWORK=off go build ./...
GOWORK=off go test ./...

cd "$WORKSPACE/smailnail"
GOWORK=off go build ./...
GOWORK=off go test -tags sqlite_fts5 ./...
```

## Phase 4: Branch and Commit State

Inspect local branch state:

```bash
ggg rollout branch ttmp/.../scripts/rollout.yaml --output table
```

If the rollout config is responsible for branch commits, use:

```bash
ggg rollout branch ttmp/.../scripts/rollout.yaml --commit --yes
```

In practice, for semantic migrations, it is often better to commit manually per repo after review:

```bash
git status --short
git diff --stat
git add <intended files>
git commit -m "..."
```

## Phase 5: Push Branches and Open PRs

Dry-run:

```bash
ggg rollout push-prs ttmp/.../scripts/rollout.yaml --dry-run --output table
```

Create/update PRs and write `prs.yaml`:

```bash
ggg rollout push-prs ttmp/.../scripts/rollout.yaml --yes --output table
```

If hooks are too expensive or known to fail for unrelated reasons, only use `--no-verify-push` with a reason:

```bash
ggg rollout push-prs ttmp/.../scripts/rollout.yaml \
  --yes \
  --no-verify-push \
  --reason "local validation already run; pre-push hook executes unrelated full repo release checks"
```

Expected PR list format:

```yaml
prs:
  - https://github.com/go-go-golems/plz-confirm/pull/13
  - repo: go-go-golems/go-go-os-chat
    number: 2
  - ref: go-go-golems/go-go-app-inventory#17
```

## Phase 6: Let PR-Open Automation Start Codex

Opening the PR already triggers a Codex run in the current repository setup. Do **not** run `ggg pr codex-trigger` immediately after PR creation as a normal step; doing so creates noise and can duplicate the automatic review request.

Instead, after opening PRs, move directly to readiness watching. Use `ggg pr codex-trigger` only as a recovery tool when readiness shows that Codex did not start, appears stale/stuck, or must be retriggered after an unusual workflow failure.

Recovery trigger for the whole PR list:

```bash
ggg pr codex-trigger --file ttmp/.../scripts/prs.yaml --wait-for-auto 30s
```

Recovery trigger for one PR:

```bash
ggg pr codex-trigger https://github.com/go-go-golems/<repo>/pull/<n> --wait-for-auto 30s
```

Safety rules built into `ggg pr codex-trigger`:

- skips if Codex already appears to be running (`EYES` reaction),
- skips if a recent human `@codex review` trigger exists,
- skips if current-head Codex feedback already exists,
- skips if Codex is already satisfied for the current head,
- use `--force` only when intentionally replacing a stale/stuck run.

Dry-run before recovery if uncertain:

```bash
ggg pr codex-trigger --file ttmp/.../scripts/prs.yaml --wait-for-auto 30s --dry-run --output json
```

## Phase 7: Watch Readiness

One-shot batch check:

```bash
ggg batch ready ttmp/.../scripts/prs.yaml --output table
```

Watch until operator action is needed:

```bash
ggg batch ready ttmp/.../scripts/prs.yaml \
  --watch \
  --until actionable \
  --interval-seconds 30 \
  --timeout-seconds 1800 \
  --output table
```

Use `--until actionable` for release trains because partial readiness is useful. Exit code `5` means at least one PR is ready while others are still waiting; merge/release ready upstreams in dependency order, then continue.

Single PR detail:

```bash
ggg pr ready https://github.com/go-go-golems/<repo>/pull/<n> --findings
```

Inspect Codex comments:

```bash
ggg pr codex-comments https://github.com/go-go-golems/<repo>/pull/<n>
ggg batch codex-comments ttmp/.../scripts/prs.yaml --group-by-message
```

### Readiness Exit Codes

| Code | Meaning | Operator action |
|---:|---|---|
| 0 | Ready / all ready | Merge in dependency order |
| 1 | Waiting | Keep watching |
| 2 | Tool/API error | Inspect `gh` auth/API output |
| 3 | Current-head Codex feedback | Read and fix/respond |
| 4 | Failed checks | Open CI logs and fix |
| 5 | Batch partial readiness | Merge/release ready upstreams |
| 6 | Merge conflict / blocked mergeability | Rebase/merge base and rerun |

## Phase 8: Merge and Publish in Dependency Order

For each ready PR:

```bash
gh pr merge <n> --merge --delete-branch
```

After merge, capture merge SHA:

```bash
gh pr view <n> --json mergeCommit --jq .mergeCommit.oid
```

For Go module releases, run preflight and tag only after the source commit is on `main`:

```bash
ggg release preflight --output json
ggg release tag-patch --dry-run --yes --output json
ggg release tag-patch --yes --output json
```

If docsctl publishing is part of the repo:

```bash
ggg release watch \
  --repo go-go-golems/<repo> \
  --workflow release.yaml \
  --tag vX.Y.Z \
  --verify-docs \
  --package <package>
```

## Phase 9: Verify Post-Merge CI/CD

Single repo:

```bash
ggg run status \
  --repo go-go-golems/<repo> \
  --branch main \
  --sha <merge-sha> \
  --ignore-workflow "Secret Scanning" \
  --output table
```

Batch manifest:

```yaml
repos:
  - repo: go-go-golems/plz-confirm
    branch: main
    sha: <merge-sha-a>
  - repo: go-go-golems/go-go-os-chat
    branch: main
    sha: <merge-sha-b>
```

Watch all:

```bash
ggg batch actions ttmp/.../scripts/actions.yaml \
  --ignore-workflow "Secret Scanning" \
  --watch \
  --output json
```

Exit codes:

| Code | Meaning |
|---:|---|
| 0 | All matching runs succeeded or were intentionally ignored |
| 1 | A non-ignored workflow failed |
| 2 | Matching runs still pending |

## Phase 10: Generate a Rollout Report

```bash
ggg rollout status ttmp/.../scripts/rollout.yaml --output table

ggg rollout report ttmp/.../scripts/rollout.yaml \
  --write-to ttmp/.../reference/rollout-status-report.md
```

Attach this report to the ticket or paste it into the final handoff.

## Decision Records

### Decision 1: Use `ggg` instead of historical scripts

- **Status:** accepted
- **Context:** infra-tooling keeps older scripts under `scripts/go-go-golems/`, but current docs identify them as historical references.
- **Decision:** use installed `ggg` commands for new operator workflows.
- **Rationale:** `ggg` provides structured output, shared readiness state machine, batch handling, Codex safeguards, and release helpers.
- **Consequence:** scripts are useful for debugging historical behavior but should not be copied into new playbooks.

### Decision 2: Store PR lists in ticket scripts

- **Status:** accepted
- **Context:** batch `ggg` commands consume YAML PR lists.
- **Decision:** store `prs.yaml` under the ticket `scripts/` directory.
- **Rationale:** PR state becomes reviewable, resumable, and suitable for diary/changelog evidence.
- **Consequence:** operators should avoid ad-hoc `/tmp/prs.yaml` for long-running rollouts unless copying it back into the ticket.

### Decision 3: Use `--until actionable` for release trains

- **Status:** accepted
- **Context:** some PRs become ready before downstream PRs can merge.
- **Decision:** default to `ggg batch ready --watch --until actionable`.
- **Rationale:** partial readiness should wake the operator so upstream packages can merge/release and unblock downstream validation.
- **Consequence:** exit code `5` is expected and useful, not a failure.

## Troubleshooting

### `gh` lacks workflow scope

If PR merge fails because workflow files changed:

```bash
gh auth refresh -h github.com -s workflow
```

### Codex appears stuck

Check readiness and comments first:

```bash
ggg pr ready <pr-url> --findings
ggg pr codex-comments <pr-url>
```

Use `--force` only after confirming the previous run is stale or irrelevant.

### Downstream build passes locally but fails in CI

Rerun with:

```bash
GOWORK=off go clean -modcache  # only if you need to prove module-cache freshness
GOWORK=off go test ./...
```

Usually the issue is an unpublished upstream tag or a local workspace masking removed symbols.

### Batch actions never finds runs

Check branch/SHA correctness:

```bash
gh run list --repo go-go-golems/<repo> --branch main --limit 10
```

Then rerun `ggg run status` with the merge SHA from `gh pr view`.
